A development harness for the agentic era

Agents write the code.
You still own it.

Builder takes a feature or bug fix from request to a reviewed, tested pull request. It asks the right questions first, builds in small test-first steps, proves the result actually runs, and hands you a report to review — nothing ships until you say so.

01 · Why a harness

The model already knows how to code. It fails in six predictable ways.

Builder exists to counter each failure mode with the weakest enforcement that actually works — because every mechanism costs context and flexibility.

FAILURE 1

Misalignment

Builds the wrong thing — you asked for X, meant Y.

↳ counter: grill before building; intent read back to you; the spec is a contract.

FAILURE 2

Laziness & self-preference

Declares "done" early and grades its own work kindly.

↳ counter: fresh-context review agents — the author never judges its own completion.

FAILURE 3

Unverified claims

"Tests pass" — without ever running them.

↳ counter: deterministic gates: real command + exit code + expected test count, shown in the transcript.

FAILURE 4

Context rot

Long sessions compact; instructions and progress evaporate.

↳ counter: state lives on disk — spec, plan, and a progress ledger that outlive any session.

FAILURE 5

Scope creep

Zero-friction additions gold-plate the diff.

↳ counter: every spec has an out-of-scope section; the diff is reviewed against the spec.

FAILURE 6

Dangerous actions

Force pushes, prod damage, one bad command.

↳ counter: shipping, tracker writes, and plugin changes are human-invoke-only — the model can't trigger them.

02 · The one idea behind everything

Process vs. facts

The plugin is process — five skills, byte-for-byte identical in every repo. They never contain a real test command or branch name; they speak in canonical verbs. Each project holds the facts in a .harness/ directory: a thin indirection layer mapping each verb to the real command for this repo. The same skill drives a Bun monorepo, a Django API, and a Rust CLI without a line of difference.

03 · The pipeline

Five phases. Two human gates. Everything else is autonomous.

Humans sit at the two highest-leverage points only: plan approval (alignment is cheapest before code exists) and the final review (taste and trust boundaries). Everything between runs on its own.

▸ ALIGN Grill first — one question at a time
Reads your product one-pager, durable state, and the code; maps the data models and interfaces the change touches (read from schema, never assumed); reads your intent back to you to catch the XY problem; then grills — ≤5 questions, each with a recommended answer, exploring the code before asking anything the code can answer. Output: a spec — objective, user stories (roles from your real personas), data & interfaces, testable acceptance criteria, and a non-negotiable out-of-scope section.
▸ PLAN The file, not the chat, is the source of truth
Weighs 2–3 approaches (losers logged — rejection decisions outlive selections), then breaks the spec into small vertical slices with exact file paths and a scope guard. The plan is written to disk before asking approval, so you can edit the file directly. Approval flips its Status line to APPROVED — the one gate before autonomous work begins.
▸ BUILD Strictly test-first, one commit per slice
Per slice: failing test shown → minimal code to pass → commit staging only that slice's files → the plan's progress ledger updated in the same breath. The ledger plus git log is the durable execution record: after compaction or a new session it's trusted over conversation memory, so a finished slice is never redone. Anything noticed-but-out-of-scope goes to the report, not the diff.
▸ PROVE The author never grades its own work
Fresh-context review agents read the diff cold (a bounded fix loop — findings get fixed and re-verified by another fresh agent), the full test suite runs through the gate with its expected count checked against the recorded baseline (a dropped count is a red flag, not a pass), and the feature is driven end-to-end like a real user — screenshots, real API calls, zero new console errors.
▸ REPORT An ownership-transfer document, then a full stop
Docs are synced, then a self-contained HTML report is written — not a checklist but a teaching document: what changed and why, the rules and edge cases now in force, flow and data-model figures, a "try it yourself" section that has you reproduce, inspect, and explore the behavior, and a quiz. Every claim wears an evidence chip (✓ verified live · ◐ fixture · ○ needs a human) — unverified edges are surfaced, never hidden. The bar: you can answer "what does the system do when X?" without reading the code. Then the pipeline stops.

04 · What setup installs

Ten minutes, once per project

/builder-setup-harness detects your stack, asks only what it can't infer (how to run tests, where docs live, what the product actually is), then writes the facts layer — and proves every command it wrote by running it. An unverified skill is a liability, not an asset.

🧭 product.md — the backbone of alignment

One page: purpose, personas, success signals, non-goals. User stories take their roles from real personas, never invented ones. A feature serving no persona or crossing a non-goal gets challenged at ALIGN — before any code exists.

🧠 STATE.md — memory that survives sessions

Recorded gate baselines, architecture decisions, lessons, rejected approaches, gotchas. Later runs diff against the baseline and don't relitigate settled decisions.

🛠️ Generated project skills — yours, not the plugin's

Setup reads your repo and generates the skills the work will need — how to bring your stack up, how to drive your UI, how to hit your API — each verified by actually running it, each growing its own Gotchas section over time.

🗺️ Works in three repo shapes

Single repo, monorepo, or a multi-repo workspace (backend + frontend side by side): gates and baselines per repo, one cross-linked PR per touched repo, harness files only ever at the root.

Every file, explained

The complete reference — what each file the harness writes holds, and why it exists.

.harness/map/ — the indirection layer: canonical verbs → real commands. The heart of the design; skills never hardcode a command, they always resolve it here.
file what it maps why it exists
gates.md "run the quick / full gate", "run the build" → the real commands, each with its expected test count. The most-used mapping. Every test invocation resolves through here; the expected counts catch silent test deletions.
tracker.md "fetch the ticket", "post the spec back", "mark ready" → the real MCP tool / CLI, plus your label vocabulary. Ports the pipeline to whatever tracker you use — or records "no tracker" so shipping skips that step cleanly.
docs.md Where specs, the glossary, and ADRs live; plus a doc-sync checklist. ALIGN reads context from here and REPORT writes updates back — so docs never drift from the code.
paths.md Protected / append-only paths and forbidden actions. The scope fence — what a run must never touch (migrations, generated code, prod config).
review.md How "run the code review" resolves — pr-review-toolkit agents if installed, else a built-in fallback reviewer. Also records your default branch. Makes the review gate portable, and pins the branch review diffs are scoped against.
design.md Where components live, which library and tokens to reuse, Storybook URL. (frontend / fullstack only) What "reuse the design system, don't reinvent components" actually points at.

05 · The self-improvement loop

Failures get encoded into the system, not into longer prompts

When a run hits a gotcha, a routing test asks: would this bite in any repo? Repo-specific lessons stay local. Universal ones travel back to the plugin source — so every install inherits the fix, and the same lesson is never relearned twice.

step 1

A run hits a universal gotcha → queued in plugin-outbox.md

→
step 2

You run /builder-improve in the plugin source

→
step 3

It truth-checks the lesson, folds it into a skill or template, bumps the version

→
step 4

Auto-update carries the fix to every installed project

Why this matters: most agent setups improve by making prompts longer, which makes them worse. Builder's rule is that a real failure becomes a structural change — a Gotcha entry, a template edit, a mapping — fixed once, for every repo, forever. The harness you install today is the sum of every failure previous runs already paid for.

06 · Get started

Three commands to your first harnessed feature

1 · install (in Claude Code) 
/plugin marketplace add kpedrok/builder-plugin
/plugin install builder@builder
2 · instrument your project — once, ~10 min
/builder-setup-harness
3 · build something
/builder-feature add CSV export to the reports page

Then open the report, check the evidence, and run /builder-ship. Optional but recommended: install pr-review-toolkit — builder uses its review agents automatically when present.
